# skill 수정 이력 - 2026-04-09

> 기준 문서: `skill-modify-history.md`

## 목적
`skill-create-rule.md`의 과도한 형식 강제를 제거하고, 적용 조건 기반의 명시적 규칙으로 정리한다.

## 사건 요약
사용자는 `skill-create-rule.md`의 H1/H2 구조, Flow, PlantUML, Monitoring 규칙이 무조건 강제로 되어 있어 실제 스킬 문서에 오히려 위험하다고 지적했다.

## 문제 원인

### 1. 형식 강제와 적용 조건이 분리되지 않음
모든 스킬에 동일한 구조를 강제해 문서 목적보다 외형 준수가 앞서는 상태가 됐다.

### 2. 예외를 섹션으로 흡수하는 방식 사용
Monitoring이 불필요한 스킬에도 섹션을 만든 뒤 "해당 없음"으로 처리하게 해 규칙 의미가 흐려졌다.

### 3. 복잡도 기준 없이 표현 수단을 강제함
비선형이라는 넓은 조건만으로 PlantUML을 강제해 작은 분기까지 다이어그램으로 밀어 넣는 구조가 됐다.

## 영향
- 짧거나 규칙형인 스킬에도 불필요한 섹션이 추가될 수 있었다.
- 형식 채우기용 Flow와 Monitoring이 생겨 문서 신뢰도가 떨어질 수 있었다.
- 간단한 분기에도 PlantUML 유지비가 발생할 수 있었다.

## 이번에 수정한 내용

### 1. skill-create-rule.md
- 본문 필수 H1을 `# Must`, `# Do NOT`, `# Definition of Done`로 재정의했다.
- `# Flow`는 절차의 선후관계가 결과에 영향을 주는 경우에만 필수로 바꿨다.
- H2는 독립 주제가 2개 이상일 때만 필수로 바꾸고, 1개뿐이면 두지 않도록 수정했다.
- PlantUML은 분기 수, 병렬 단계, 재진입, 승인 주체 수 기준을 만족할 때만 필수로 바꿨다.
- Monitoring은 자동화·운영 감시 조건을 만족할 때만 필수로 바꾸고, 형식상 `해당 없음` 섹션 추가를 금지했다.
- Authoring, Done Design, Verification를 위 조건부 규칙과 일치하도록 함께 수정했다.
- 문서 자체가 Monitoring 비대상이라 `## Monitoring` 섹션을 제거해 자기 충돌을 없앴다.
- `Rule Writing`에 남아 있던 `권장` 강도 분류를 제거하고, `조건부 의무`와 `선택 가능` 분류로 교체했다.
- `Weak Compression`에 남아 있던 `의무/권장/금지` 표현도 `의무/조건부 의무/금지/선택 가능`으로 교체했다.
- `Definition of Done > Verification`에 항상 필수인 H1/H2 누락 여부를 먼저 확인하는 검증 항목을 추가했다.
- `Definition of Done > Verification`의 검증 순서를 절대 필수 누락 검사 → 조건부 필수 적용 검사 → 기타 규칙 검사 순서로 재정렬했다.
- `Definition of Done > Verification`의 `확인 가능하다`, `충돌하지 않는다` 같은 판정 여지 표현을 `확인한다` 기반 문장으로 교체했다.
- 이후 `선택 가능`도 판단 슬롯이라는 지적을 반영해, `Rule Writing`의 강도 분류를 `의무`, `조건부 의무`, `금지`만 남기도록 다시 정리했다.
- `Weak Compression`의 강도 분류도 같은 체계에 맞춰 `의무/조건부 의무/금지`로 다시 축소했다.
- `Structure`에 남아 있던 `H1을 추가할 수 있다` 표현도 `H1을 추가한다`로 바꿔 마지막 재량 표현을 제거했다.
- `Section Design`의 출력 예시 규칙도 조건 기반으로 다시 정의해, 예시를 넣는 경우와 넣지 않는 경우를 모두 명시했다.
- `Structure`의 `표현하기 어려운`은 `독립 규칙의 범위가 분리되지 않으면`으로 교체해 H1 추가 조건을 더 기계적으로 정의했다.
- `Rule Writing`의 `여러 행동이 필요하면`은 `동사 또는 행위 대상이 2개 이상이면`으로 교체해 항목 분리 조건을 더 명시적으로 정의했다.
- 이후 더 강한 판단 제거 기준에 맞춰, H1 추가 조건은 `위 필수 H1에 포함되지 않는 규칙이 있으면`으로 다시 단순화했다.
- 항목 분리 규칙도 `하나의 항목은 하나의 행동만 포함한다`로 다시 단순화했다.
- `Structure`에 H1 순서를 명시해 `Verification`의 H1 순서 검증과 원문 규칙이 충돌하지 않도록 맞췄다.
- `독립 주제` 판정 기준도 적용 대상, 실행 시점, 실패 처리 기준으로 명시해 H2 사용 여부를 더 기계적으로 결정하게 했다.
- 별도 H1 추가 조건도 목적, 적용 대상 혼합, 실행 시점 혼합 기준으로 다시 구체화했다.
- `Frontmatter`의 `description` 규칙은 `대상, 적용 시점 또는 조건, 수행 행동`을 포함하도록 바꿔 검증 기준을 더 명확히 했다.
- 이후 규칙 증식이 과도하다고 판단해, 위 과세분화 규칙 다수는 되돌리고 stage 1 수준의 구조 개편만 남기도록 다시 정리했다.
- 되돌린 항목은 H1 순서 강제, `독립 주제` 판정 기준, 별도 H1 추가 조건 세분화, `description`의 과도한 검증형 표현, `Verification`의 세밀한 판정 문구다.
- 반복 수정 과정의 과적합 중단 규칙은 `skill-create-rule.md`의 책임 범위를 벗어난다고 판단해 다시 제거했다.

## 재발 방지 규칙
1. 형식 규칙은 `의무`, `조건부 의무`, `금지` 중 하나로만 정의한다.
2. `권장`, `선택 가능` 같은 표현으로 구조 규칙이나 강도 분류를 정의하지 않는다.
3. 문서 구조를 강제할 때는 적용 조건과 비적용 조건을 같은 섹션에서 함께 닫는다.
4. 특정 표현 수단을 강제할 때는 복잡도 기준을 먼저 정의한다.
5. 스킬 문서 자체도 수정된 구조 규칙에 자기 일관적으로 맞는지 확인한다.
