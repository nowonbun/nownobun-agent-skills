## 문제 원인

### 1. 스킬 규칙 위반

- `markdown-safe-writing`에는 Windows PowerShell 경유 한글 heredoc/파이프 입력을 피하라는 규칙이 이미 있었다.
- 실제 작업에서는 그 규칙을 지키지 않고 한글 본문을 직접 PowerShell 파이프로 전달했다.

### 2. 검증 절차 부족

- `read_text(encoding='utf-8')`가 성공한다는 이유만으로 문서가 정상이라고 잘못 판단했다.
- 실제로는 한글이 이미 `?`로 바뀐 상태여도 UTF-8 읽기는 성공할 수 있다.

### 3. 교차 리뷰 범위 부족

- Claude 리뷰 프롬프트를 수치 정합성 검토로만 제한했다.
- 따라서 인코딩 손상, `?` 치환, 한글 문장 보존 여부는 리뷰 범위에 포함되지 않았다.

## 재발 방지 규칙

- Windows PowerShell에서 한글 본문을 heredoc, 파이프, 표준입력으로 직접 Python이나 다른 프로세스에 전달하지 않는다.
- Markdown 문서 수정은 가능하면 `apply_patch`를 1순위로 사용한다.
- 부득이하게 스크립트로 문서를 쓸 때는 Base64, 유니코드 이스케이프, 기존 파일 복사 후 부분 치환 등 ASCII-safe 방식만 사용한다.
- 한글 문서 작성 후에는 아래를 모두 확인한다.
  - `read_text(encoding='utf-8')` 성공 여부
  - 비정상적인 `?` 반복 여부
  - 대표 한글 문장 2개 이상이 실제로 보존됐는지
- Claude 교차 리뷰를 요청할 때는 수치 검토와 별도로 **인코딩/문자 깨짐 여부 확인**을 명시한다.
